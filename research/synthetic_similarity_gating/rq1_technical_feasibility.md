# RQ1: Technical Feasibility of Synthetic-to-Task Comparison

## Overview
This report investigates the technical feasibility of using the local embedding model `all-MiniLM-L6-v2` to perform real-time validation of synthetic data generated by the Architect agent. By comparing the synthesized example against the original user task, we can ensure semantic alignment and prevent domain drift before the data is used for downstream processing.

---

## 1. Embedding Reuse: Implementation in Python

Since the `all-MiniLM-L6-v2` model is already utilized for RAG (Retrieval-Augmented Generation) within the system, we can reuse the same instance to compute similarity for synthetic examples. This avoids the overhead of loading a second model into memory.

### Implementation Logic
The following Python implementation utilizes the `sentence-transformers` library to compute the cosine similarity between the user task and the synthetic output.

```python
from sentence_transformers import SentenceTransformer, util
import torch

class SyntheticValidator:
    def __init__(self, model_name='all-MiniLM-L6-v2'):
        # Reuse the existing model instance
        self.model = SentenceTransformer(model_name)

    def compute_similarity(self, task_text: str, synthetic_text: str) -> float:
        """
        Computes the cosine similarity between the user task and synthetic data.
        """
        # Encode both texts into 384-dimensional embeddings
        embeddings = self.model.encode([task_text, synthetic_text], convert_to_tensor=True)
        
        # Calculate cosine similarity
        # util.cos_sim returns a matrix; we take the top-right value [0][1]
        similarity = util.cos_sim(embeddings[0], embeddings[1])
        
        return float(similarity.item())

# Usage Example:
# validator = SyntheticValidator()
# score = validator.compute_similarity("Extract patient symptoms", "Patient shows signs of fever")
```

### Efficiency Considerations
*   **Latency:** `all-MiniLM-L6-v2` is a 6-layer transformer with 22.7M parameters. Encoding a single synthetic sentence typically takes **< 10ms** on a modern CPU, making it feasible for real-time gating.
*   **Memory:** The model takes approximately **80MB** of RAM, which is already budgeted in the current architecture.

## 2. Similarity Thresholds: Defining "Rejection"

The system currently uses **0.65** as a gate for RAG retrieval (ensuring retrieved documents are relevant). For synthetic data comparison, the threshold should be **more strict**.

### Recommended Thresholds
| Threshold | Status | Interpretation |
| :--- | :--- | :--- |
| **> 0.85** | **Excellent** | High-fidelity synthetic example; near-perfect semantic match. |
| **0.70 – 0.85** | **Acceptable** | Valid semantic alignment; likely covers an edge case or paraphrase. |
| **0.60 – 0.70** | **Warning** | Potential drift. Example may be too generic or tangential. |
| **< 0.60** | **Reject** | High likelihood of hallucination or domain mismatch. |

**Rationale:** Synthetic data generated by the Architect is intended to be a *gold-standard* representation of the task. If the similarity falls below 0.70, the Architect has likely failed to capture the specific nuances of the user's instructions, whereas RAG can afford a lower threshold (0.65) because it is searching through a noisy corpus.

---

## 3. Domain Alignment: Detecting Drift

Domain drift occurs when the Architect generates an example that is semantically valid but belongs to the wrong industry or context (e.g., a "Legal" example for a "Medical" task).

### Detection Methods
1.  **Centroid Comparison:** 
    *   Maintain a "Domain Centroid" (average vector) for different domains (Medical, Legal, Finance).
    *   If the synthetic vector is closer to a non-target centroid than the task vector, flag it as drift.
2.  **Top-K Nearest Neighbor Check:** 
    *   Retrieve the top 5 most similar "known-good" examples from the RAG store.
    *   If the synthetic example's average similarity to these neighbors is significantly lower than the task's similarity to them, drift is detected.
3.  **Out-of-Distribution (OOD) Gating:**
    *   Use the `all-MiniLM-L6-v2` embedding to calculate the Mahalanobis distance from a baseline distribution of valid tasks.

## 4. Mathematical Formula for Validation: The 'Audit Score'

To provide a single metric for the Auditor agent to decide whether to accept or reject a synthetic example, we define the **Audit Score ($S_{audit}$)**.

### The Formula
The Audit Score is a weighted harmonic mean of semantic similarity and structural integrity:

$$S_{audit} = \frac{2 \cdot (S_{sim} \cdot S_{struct})}{S_{sim} + S_{struct}} \times C_{domain}$$

Where:
*   **$S_{sim}$**: Normalized Cosine Similarity $[0, 1]$.
    *   $S_{sim} = \frac{\cos(\mathbf{v}_t, \mathbf{v}_s) + 1}{2}$
*   **$S_{struct}$**: Structural Alignment Score $[0, 1]$.
    *   Calculated based on schema validation (JSON match, field presence).
*   **$C_{domain}$**: Domain Confidence Coefficient.
    *   1.0 if domain aligns; 0.5 if drift is suspected; 0.0 if domain mismatch is confirmed.

### Decision Logic
*   **Accept:** $S_{audit} \geq 0.75$
*   **Refine:** $0.60 \leq S_{audit} < 0.75$ (Request Architect to re-generate)
*   **Reject:** $S_{audit} < 0.60$ (Abort pipeline)

## Conclusion
Technical feasibility is **High**. The reuse of `all-MiniLM-L6-v2` provides a low-latency, memory-efficient mechanism for gating. Implementing a stricter threshold of **0.70** for synthetic rejection (compared to 0.65 for RAG) ensures high data quality while the proposed **Audit Score** formula offers a robust multi-dimensional validation metric.
