# Engine Integration: Two-Pass Champion Pipeline

This document outlines the technical design for refactoring the **Two-Pass-NI (Negative Invariants)** strategy into the modular framework. It focuses on the transition from a monolithic baseline to a standardized `IExtractionEngine` that interacts with the `ARCHITECT` component.

## 1. Refactoring Two-Pass-NI into `IExtractionEngine`

The existing logic in `src/gensie/baseline.py` implements a sequential two-pass approach. To integrate this into the modular pipeline, we define a standardized interface that allows the `ARCHITECT` agent to inject refined extraction logic.

### Standardized Interface: `IExtractionEngine`

The engine is responsible for the execution of the extraction passes. It consumes instructions generated by the `ARCHITECT`.

```python
from typing import Any, Dict, List, Optional, Protocol
from pydantic import BaseModel

class ExtractionResult(BaseModel):
    """Container for multi-pass extraction results and telemetry."""
    data: Dict[str, Any]
    pass1_output: Dict[str, Any]
    usage: Dict[str, int]  # prompt_tokens, completion_tokens
    latency_seconds: float
    model_name: str

class IExtractionEngine(Protocol):
    """Interface for extraction engines within the Two-Pass framework."""
    async def extract(
        self, 
        text: str, 
        schema: Dict[str, Any], 
        instructions: Optional[Dict[str, Any]] = None
    ) -> ExtractionResult:
        """
        Executes the extraction process. 
        In the Two-Pass-NI implementation, this performs:
        1. Pass 1: Initial Discovery/Extraction
        2. Pass 2: Invariant-based Auditing/Refinement
        """
        ...
```

### Implementation Details: `TwoPassChampionEngine`

1.  **Pass 1 (Drafting):** The engine uses the `schema` and any `positive_hints` from the `ARCHITECT` to perform a high-recall extraction.
2.  **Pass 2 (Auditing):** The engine feeds the `text`, `schema`, and `pass1_output` back to the model, applying the `negative_invariants` (refined constraints) to prune hallucinations and enforce logical consistency.

```python
# Conceptual implementation snippet
class TwoPassChampionEngine:
    async def extract(self, text, schema, instructions):
        # Step 1: Draft
        draft = await self.llm.generate(
            prompt=self.prompter.draft(text, schema, instructions.get("hints"))
        )
        
        # Step 2: Audit (Negative Invariants)
        final = await self.llm.generate(
            prompt=self.prompter.audit(
                text, 
                schema, 
                draft, 
                invariants=instructions.get("negative_invariants")
            )
        )
        return ExtractionResult(data=final, pass1_output=draft, ...)
```

## 2. Unification of the `Null Rule` Invariant

The `Null Rule` is a critical safety invariant found in the `Two-Pass-NI` research. It prevents the model from generating "filler" data or hallucinations when the target entities are absent from the source text.

### Global Pipeline Integration
To ensure consistency across the pipeline, the `Null Rule` is treated as a **Global Invariant**:

-   **Definition:** "If the source text contains no information relevant to the schema or a specific field, the output for that field must be `null` (or `{}` for the entire object)."
-   **Architect Integration:** The `ARCHITECT` agent automatically injects this rule into the `negative_invariants` payload for every `IExtractionEngine` call.
-   **Enforcement:** In the second pass, the "Auditor" is explicitly prompted to check for "Null Violations"—instances where the first pass extracted entities that do not actually exist in the text.

## 3. Baseline Latency and Token Cost

Based on the performance data in `research/pipeline_ranking_report.md`, the Two-Pass-NI strategy incurs higher costs in exchange for superior accuracy.

### Estimated Metrics (per execution)

| Metric | Single-Pass Baseline | Two-Pass-NI (Modular) | Variance |
| :--- | :--- | :--- | :--- |
| **Latency (Mean)** | 2.1s | 4.2s | +100% |
| **Input Tokens** | ~1,100 | ~2,500 | +127% |
| **Output Tokens** | ~200 | ~250 | +25% |
| **Success Rate (F1)** | 0.78 | 0.92 | +14% |

**Cost Breakdown:**
-   **Latency:** Primarily driven by the sequential nature of the two calls. Future optimizations may include "Small Model Draft -> Large Model Audit" to reduce costs.
-   **Input Tokens:** The second pass requires re-sending the source text and the draft output, which nearly doubles the input token count.
-   **Value Proposition:** The ~14% increase in F1 score justifies the cost in high-precision domains (e.g., Cultural/Environmental data extraction) where hallucinations are costly.
